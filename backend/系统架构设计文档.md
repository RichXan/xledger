# Xledger v1 后端系统架构设计文档

> 版本：v1.0（设计评审稿）  
> 日期：2026-03-19  
> 适用范围：`v1 核心链路 + 可选增强（AI/Shortcuts）`

---

## 1. 文档目标与范围

本文件用于给出 Xledger 后端的架构主设计，重点回答三个问题：

1. 系统由哪些核心单元构成，边界在哪里。
2. 关键业务口径如何在各单元间保持一致。
3. 可选增强（n8n + LLM + Apple Shortcuts）如何接入且不影响核心可用性。

本文件不展开到字段级实现细节与伪代码；业务域落地方案见 `backend/技术方案文档.md`。

---

## 2. 架构总览

### 2.1 上下文架构

```text
React PWA / API Client
        |
        v
  Go API (Gin)
   |- Auth Context
   |- Accounting Context
   |- Classification Context
   |- Reporting Context
   |- Portability Context
        |
        +--> PostgreSQL (主数据)
        +--> Redis (验证码/黑名单/限流)
        +--> SMTP Server (验证码邮件发送)
        +--> n8n (可选AI编排)
        +--> Google OAuth Provider
```

### 2.2 分层视角

- 接入层：HTTP 路由、鉴权、参数校验、错误映射、限流。
- 应用层：按用例组织（认证、交易、统计、导入导出、PAT 管理）。
- 领域层：沉淀强规则（总资产口径、转账双边一致性、分类归档语义）。
- 基础设施层：PostgreSQL、Redis、外部适配器（SMTP/n8n/OAuth）。

### 2.3 核心非功能目标

- 一致性优先：账务数据正确性高于可选增强能力。
- 可降级：外部依赖（n8n/LLM）异常不阻断核心记账链路。
- 自托管友好：最小依赖可运行，配置边界清晰。

## 3. 领域边界与职责

### 3.1 Auth Context（认证与会话）

- 负责：邮箱验证码登录、Google OAuth 登录、JWT/Refresh 生命周期、登出黑名单。
- 依赖：Redis（验证码、黑名单）、SMTP（邮件发送）、OAuth Provider。
- 不负责：业务数据授权模型之外的财务领域规则。

### 3.2 Accounting Context（账本/账户/交易核心）

- 负责：账本管理、账户管理、收入/支出/转账写入、余额回算与一致性。
- 关键点：转账采用双边交易模型，必须同事务提交。
- 不负责：图表聚合展示逻辑（交由 Reporting Context）。

### 3.3 Classification Context（分类与标签）

- 负责：默认分类初始化、分类两级结构管理、标签管理。
- 关键点：分类被交易引用后删除需归档，不做物理删除。
- 不负责：导入映射推荐（可选增强能力，交由 n8n 流程）。

### 3.4 Reporting Context（统计与看板）

- 负责：概览指标、趋势统计、分类聚合。
- 关键点：`total_assets` 固定全账户口径，不受账本筛选影响。
- 不负责：交易写入与一致性事务。

### 3.5 Portability Context（导入导出与开放能力）

- 负责：CSV/JSON 导出、CSV 导入确认链路、PAT 生命周期。
- 关键点：导出必须保留历史分类名称语义。
- 不负责：用户认证令牌签发（PAT 不可替代 auth token）。

### 3.6 Automation Context（可选增强）

- 负责：CSV AI 映射建议、Apple Shortcuts AI 快速记账编排。
- 关键点：作为外挂能力存在，默认可关闭。
- 失效策略：不可用时返回明确错误，不污染核心账务数据。

## 4. 关键业务口径与一致性规则

### 4.1 资产与统计统一口径

- 账户余额：`initial_balance + SUM(account transactions)`。
- 总资产：所有账户当前余额总和。
- `account_id = NULL` 交易：计入收支/分类/趋势统计，不计入总资产。
- 账本筛选：影响收支类统计，不影响总资产指标。

### 4.2 转账一致性

- 转账由两条交易组成，共享 `transfer_pair_id`。
- 创建、编辑、删除必须成对处理且同事务完成。
- 默认统计视图排除转账内部对冲，避免重复计入收支。

### 4.3 分类历史语义

- 默认分类按“用户空间拷贝”初始化。
- 已被引用分类删除时归档，历史交易引用保留。
- 统计与导出按历史分类名称展示，保障可解释性。

## 5. 核心数据流

### 5.1 登录链路（邮箱验证码）

1. 客户端请求发送验证码。
2. Go API 做频控并写 Redis。
3. Go API 通过 SMTP 账号密码直接发送验证码邮件。
4. 用户提交验证码，验证通过后签发 Access/Refresh。
5. 首次登录自动创建用户与默认总账本。

### 5.2 手动记账链路

1. 客户端提交交易。
2. Accounting Context 校验账本、账户、分类有效性。
3. 在事务中写交易并更新余额相关数据。
4. Reporting Context 读取时按统一口径聚合。

### 5.3 可选快捷记账链路（Shortcuts）

1. Apple Shortcuts 调用 n8n Webhook。
2. n8n 校验 PAT 并调用 LLM 解析文本。
3. n8n 调用 Go API 创建交易。
4. 账户提示无法唯一匹配时，安全降级 `account_id = NULL`。

## 6. 跨域能力设计

### 6.1 安全与隐私

- Access Token 15 分钟，Refresh Token 7 天。
- Refresh 登出后写入 Redis 黑名单。
- PAT 仅保存哈希，默认 90 天有效期，可撤销。
- 敏感配置全部走环境变量，不在代码或文档示例中存明文。

### 6.2 可观测性

- 日志：结构化日志，贯穿 `request_id`。
- 指标：接口延时、错误率、导入导出耗时、外部依赖可用性。
- 告警：认证失败激增、验证码邮件发送失败率异常、导入失败率异常、n8n 连续不可用。

### 6.3 配置与部署

- 核心 Compose：Go API、PostgreSQL、Redis、前端。
- n8n 不强绑定到核心 Compose，可通过 `N8N_BASE_URL` 对接外部实例。
- AI 开关默认关闭，按环境逐级开启。

## 7. 可选增强接入策略

### 7.1 设计原则

- 不改变核心域边界，不绕过核心校验。
- 不引入“半成功账务写入”。
- 所有失败可观察、可回溯、可重试。

### 7.2 失败隔离

- n8n/LLM 异常：只影响可选能力，不影响手动记账。
- OAuth Provider 异常：仅影响 Google 登录，不影响邮箱验证码链路。
- AI 推荐失效：保留手动映射与人工纠错路径。

---

## 8. 风险与演进

### 8.1 主要风险

- 统计口径漂移导致“看板与明细不一致”。
- 转账与余额回算复杂度上升导致账实不符风险。
- 自托管环境差异影响部署成功率。

### 8.2 架构层缓解策略

- 在领域层固化口径规则，禁止在接口层重复定义。
- 转账全链路事务化并建立专项回归验证。
- 可选增强严格外挂化，主链路不依赖外部 AI 服务。

### 8.3 后续演进边界

- v1.1 可扩展离线同步、多币种展示、深色模式。
- v2.0 再考虑协作账本、Webhook、AI 洞察。
