# Xledger v1 后端技术方案文档（按业务域拆分）

> 版本：v1.0（设计评审稿）  
> 日期：2026-03-19  
> 适用范围：`v1 核心链路 + 可选增强（AI/Shortcuts）`

---

## 1. 文档定位

本文件承接 `backend/系统架构设计文档.md`，按业务域给出技术落地方案。每个业务域使用同一模板描述：

- 目标
- 核心流程
- 接口范围
- 数据与约束
- 异常与降级
- 验收要点

当前深度为“设计评审级”，用于：

1. 统一团队实现认知。
2. 支撑后续实现计划拆解。
3. 作为联调与验收口径依据。

## 2. 全局约定

### 2.1 API 风格

- 资源导向 REST 风格，语义清晰、边界明确。
- 同一能力仅保留一条主路径，避免多入口造成口径分裂。
- 统一响应结构：成功返回业务数据，失败返回标准错误对象。

### 2.2 认证与授权

- 会话认证：Access Token + Refresh Token。
- 开放能力：PAT（仅业务 API，不可用于签发/刷新认证令牌）。
- 业务 API 同时支持用户会话与 PAT 鉴权，但路径权限需显式区分。

### 2.3 错误处理规范

- 错误返回结构：`code`、`message`、`request_id`、`details?`。
- 所有外部依赖错误（SMTP/OAuth/n8n/LLM）映射为可识别错误码，不泄露内部信息。
- 可重试错误与不可重试错误在语义上清晰区分。

### 2.4 事务与一致性

- 涉及账务金额的写入必须事务化。
- 转账双边交易需同事务提交/回滚。
- 编辑与删除交易需要触发余额回算并保证原子性。

## 3. 业务域方案

### A. 认证与会话域（Auth Context）

#### A.1 目标

- 提供低摩擦登录体验（邮箱验证码 + Google OAuth）。
- 将多种登录入口收敛到统一会话模型。
- 保证登出后 Refresh Token 不可继续使用。

#### A.2 核心流程

1. 邮箱验证码登录：发送验证码 -> 验证 -> 签发 Token。
2. Google 登录：OAuth 授权回调 -> 账号关联/创建 -> 签发 Token。
3. 首次登录初始化：创建用户、默认总账本、默认分类模板。
4. 退出登录：Refresh Token 加入黑名单，客户端清理本地 Token。

#### A.3 接口范围

- `POST /api/auth/send-code`
- `POST /api/auth/verify-code`
- `GET /api/auth/google`
- `GET /api/auth/google/callback`
- `POST /api/auth/refresh`
- `POST /api/auth/logout`
- `GET /api/auth/me`

#### A.4 数据与约束

- 验证码 6 位数字，10 分钟有效。
- 同邮箱 60 秒限发；同 IP 每小时发送上限控制。
- Access Token 15 分钟；Refresh Token 7 天。
- Refresh Token 黑名单存在 Redis，支持过期清理。
- SMTP 通过环境变量配置（`SMTP_HOST`、`SMTP_PORT`、`SMTP_USER`、`SMTP_PASS`、`SMTP_FROM`）。

#### A.5 异常与降级

- SMTP 邮件发送失败：返回可重试错误，提示稍后再试。
- OAuth 外部异常：不影响验证码登录链路。
- Redis 异常时：登录入口应返回明确系统不可用错误。

#### A.6 验收要点

- 新用户首次登录后可立即记账（无需先建账户）。
- 登出后 Refresh Token 无法继续刷新。
- 双登录方式签发令牌语义一致。

### B. 账本/账户/交易核心域（Accounting Context）

#### B.1 目标

- 支持收入、支出、转账完整闭环。
- 支持“无账户也可记账”，同时保证有账户时余额准确。
- 保证交易变更后统计与余额回算一致。

#### B.2 核心流程

1. 普通交易（收入/支出）：校验 -> 入库 -> 更新相关聚合。
2. 转账交易：创建双边交易记录并绑定同一 `transfer_pair_id`。
3. 交易编辑：重新校验并回算受影响账户余额与统计。
4. 交易删除：删除记录并回算；转账删除需双边同步。

#### B.3 接口范围

- `GET /api/ledgers`
- `POST /api/ledgers`
- `PATCH /api/ledgers/{id}`
- `DELETE /api/ledgers`
- `GET /api/accounts`
- `POST /api/accounts`
- `PATCH /api/accounts/{id}`
- `DELETE /api/accounts`
- `GET /api/transactions`
- `POST /api/transactions`
- `PATCH /api/transactions/{id}`
- `DELETE /api/transactions`

#### B.4 数据与约束

- `Transaction.ledger_id` 必填，默认总账本。
- `Transaction.account_id` 可空。
- 转账必须同时指定转出与转入账户，不允许空账户转账。
- 账户属于用户，不属于账本。
- 用户仅有一个默认总账本，且不可删除。

#### B.5 异常与降级

- 转账任一边写入失败：整体回滚。
- 编辑导致账户不合法或账本不合法：拒绝提交并返回可读错误。
- 回算失败：事务失败，禁止部分成功。

#### B.6 验收要点

- 普通交易创建后，相关统计即时可见。
- 转账双边交易始终成对存在。
- 编辑/删除后余额、流水、统计保持一致。

---

### C. 分类/标签与查询域（Classification Context）

#### C.1 目标

- 支持用户可编辑的分类体系（两级结构）与标签体系。
- 在分类变化后保留历史交易与报表语义。

#### C.2 核心流程

1. 用户注册时复制默认分类模板到用户空间。
2. 用户可新增/编辑/删除/归档分类。
3. 分类被历史交易引用时，删除动作转化为归档动作。
4. 交易查询支持账本、账户、分类、标签、时间范围筛选。

#### C.3 接口范围

- `GET /api/categories`
- `POST /api/categories`
- `PATCH /api/categories/{id}`
- `DELETE /api/categories`
- `GET /api/tags`
- `POST /api/tags`
- `PATCH /api/tags/{id}`
- `DELETE /api/tags`

#### C.4 数据与约束

- 分类结构限定为“一级 + 一级子分类”。
- 分类支持 `source=system/custom` 标识来源。
- 分类支持归档状态，归档后不可用于新交易选择。
- 标签可多选，交易与标签为多对多关系。

#### C.5 异常与降级

- 删除已引用分类：自动转归档并提示用户。
- 查询过滤参数冲突：返回参数校验错误。
- 标签重复创建：返回冲突错误并引导复用已有标签。

#### C.6 验收要点

- 历史交易在分类删除/归档后仍可正确展示原分类名称。
- 筛选性能满足 PRD 中 1k 数据量目标。

---

### D. 统计与 Dashboard 域（Reporting Context）

#### D.1 目标

- 为首页与统计页面提供统一口径数据。
- 保证“总资产”与“收支统计”口径边界清晰。

#### D.2 核心流程

1. 概览统计：总资产、本月收入、本月支出、本月结余。
2. 趋势统计：按时间窗口聚合收入/支出。
3. 分类统计：按分类聚合支出，兼容历史分类名称。

#### D.3 接口范围

- `GET /api/stats/overview`
- `GET /api/stats/trend`
- `GET /api/stats/category`

#### D.4 数据与约束

- `total_assets` 不受账本筛选参数影响。
- 收入/支出/结余支持账本与时间范围筛选。
- `account_id = NULL` 交易计入收支与分类聚合，不计入总资产。
- 转账对冲记录在默认收支口径中排除。

#### D.5 异常与降级

- 聚合查询超时：返回可重试错误码，不返回不完整数据。
- 参数非法（时间范围反转等）：快速失败并提示修正。

#### D.6 验收要点

- 统计数据与交易明细抽样核对一致。
- Dashboard 查询性能满足 PRD 指标目标。

---

### E. 导入导出与开放能力域（Portability Context）

#### E.1 目标

- 支持用户数据可携带（导入/导出）。
- 提供可控开放能力（PAT + 业务 API）。

#### E.2 核心流程

1. CSV 导入：上传 -> 字段映射预览 -> 用户确认 -> 入库与结果回执。
2. 数据导出：按格式与时间范围导出 CSV/JSON。
3. PAT 管理：生成、列出、撤销，限制作用范围。

#### E.3 接口范围

- `POST /api/import/csv`
- `POST /api/import/csv/confirm`
- `GET /api/export`
- `GET /api/personal-access-tokens`
- `POST /api/personal-access-tokens`
- `DELETE /api/personal-access-tokens/{id}`

#### E.4 数据与约束

- 导入去重基于（金额 + 日期 + 描述）三元组精确匹配。
- 导出需保留交易创建时分类名称语义。
- PAT 仅保存哈希值，不保存明文。
- PAT 默认有效期 90 天，可缩短不可放宽为永久。

#### E.5 异常与降级

- 导入部分失败：提供逐项失败原因，成功部分保留。
- 导出失败：允许重试，不产生半成品状态。
- PAT 被撤销或过期：统一鉴权失败响应。

#### E.6 验收要点

- 10k 交易规模导出可在目标时限内完成。
- 导入结果统计（成功/跳过/失败）可解释且可追溯。
- PAT 不可访问认证签发/刷新接口。

### F. 可选增强域（Automation Context）

#### F.1 目标

- 在不影响核心发布门禁的前提下提供 AI 增强效率能力。

#### F.2 范围

- CSV AI 字段映射建议（可选）。
- Apple Shortcuts + n8n + LLM 快速记账（可选）。

#### F.3 核心流程

1. 快捷指令调用 `POST {n8n_url}/webhook/quick-entry`。
2. n8n 校验 PAT，调用 LLM 解析文本。
3. n8n 通过业务 API 创建交易。
4. `account_hint` 不唯一时降级 `account_id = NULL`。

#### F.4 接口边界

- Go 后端不新增面向终端用户的 `/api/quick-entry`。
- AI 编排入口由 n8n Webhook 对外暴露。

#### F.5 异常与降级

- LLM 不可用：返回明确失败，不创建错误交易。
- n8n 不可用：仅该增强功能不可用，核心手动记账不受影响。
- PAT 失效：快捷链路鉴权失败并可审计。

#### F.6 验收要点

- 自然语言样本在交易类型与金额提取上达到 PRD 目标。
- 失败场景不产生账务脏数据。

## 4. 横切实现策略

### 4.1 日志与追踪

- 全链路透传 `request_id`。
- 鉴权失败、导入失败、外部依赖失败记录结构化事件。
- 可选增强链路需记录外部调用阶段与耗时，便于定位故障。

### 4.2 监控与告警

- 核心指标：接口成功率、P95 延迟、错误码分布。
- 业务指标：记账成功率、导入成功率、导出耗时、Token 刷新成功率。
- 外部依赖指标：SMTP、n8n、OAuth、LLM 调用可用性。

### 4.3 测试策略（评审级）

- 单元测试：金额口径、分类状态转换、PAT 生命周期规则。
- 集成测试：认证流程、转账事务、导入导出、统计一致性。
- 回归测试：可选增强开关启停、外部依赖异常下核心链路可用性。

### 4.4 发布与回滚

- 发布顺序：迁移 -> 应用更新 -> 开关可选增强。
- 回滚顺序：先关闭可选增强，再回退应用，最后处理迁移兼容。
- 对账策略：若出现账务异常，以交易明细与账户余额校验任务为准执行人工修复。

## 5. 里程碑映射

- M1（Foundation）：认证、默认账本、基础环境、SMTP 邮件发送能力。
- M2（Core Ledger）：交易/转账、分类标签、回算。
- M3（Reporting & Portability）：看板统计、导入导出、PAT、OpenAPI。
- M4（Optional）：AI 映射建议、Shortcuts 快记与部署文档。

该映射用于计划阶段的任务拆分，不改变本文件按业务域组织的结构。

---

## 6. 结论

本方案通过“架构主文档统一边界 + 技术方案按业务域落地”的方式，确保：

1. 核心账务口径统一且可验证。
2. 业务实现边界清晰，便于并行开发。
3. 可选增强可插拔，不阻塞 v1 核心交付。
